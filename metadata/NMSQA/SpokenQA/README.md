# NMSQA (Natural Multi-speaker Spoken Question Answering)

## Overview
**NMSQA** (Natural Multi-speaker Spoken Question Answering) is a textless spoken question answering dataset based on the **SQuAD** (Stanford Question Answering Dataset). The dataset is designed to evaluate models on spoken question answering tasks where both the context passage and questions are provided in audio format, requiring models to process speech directly without relying on text transcriptions. Audio is synthesized using natural multi-speaker text-to-speech systems, with passages sampled at **22.05 kHz** (train/dev) or **16 kHz** (test), stored in **MP3** or **WAV** format. Each passage audio is paired with multiple questions, making this a challenging benchmark for textless speech understanding and reasoning.

## Supported Tasks
1. **Spoken Question Answering**

## Data Format

Each sample is stored as a JSON entry with the following fields:

| Field | Description |
|------|-------------|
| `id` | Unique question ID (format: `{squad_id}_Q{num}`) |
| `path` | Path to audio file containing the context passage |
| `sampling_rate` | Audio sampling rate (22050 or 16000 Hz) |
| `duration` | Audio duration (seconds) |
| `dataset` | Source dataset |
| `question` | Text of the question |
| `answer` | Text of the answer |

**Note:** Multiple questions may reference the same passage audio file. Questions are distinguished by appending `_Q{num}` to the original SQuAD ID.

## Task Usage

### 1. Spoken Question Answering
- **Input:** Spoken context passage (audio) + Question text
- **Target field:** `answer` (text answer extracted from spoken passage)

**Task Description:** Given a spoken passage and a text question about that passage, the model must extract or generate the correct answer. This evaluates the model's ability to understand and reason over spoken content without relying on ASR transcriptions.

---

## Label Space

*This task does not have a predefined label space - it generates open-ended text answers based on extractive or abstractive question answering over spoken passages.*

---

## Audio Characteristics

### Sampling Rates
- **Train/Dev**: 22.05 kHz (MP3 format)
- **Test**: 16 kHz (WAV format)

### Audio Format
- **Train/Dev**: MP3 compressed audio
- **Test**: WAV uncompressed audio

### Duration Range
- Passage durations vary from ~15 seconds to ~60 seconds
- Average passage duration: ~30-40 seconds
- Passages contain full context paragraphs from SQuAD

### Speech Synthesis
- Generated using natural multi-speaker TTS systems
- Multiple speakers provide diversity in voice characteristics
- High-quality synthetic speech designed to sound natural

---

## Dataset Construction

### Source Data
Based on the **SQuAD 1.1** question answering dataset:
- Context passages from Wikipedia articles
- Questions generated by crowdworkers
- Extractive answers (text spans from passages)

### Spoken Audio Generation
1. **Text-to-Speech Synthesis**: Context passages converted to speech using multi-speaker TTS
2. **Natural Variation**: Multiple speakers used to provide acoustic diversity
3. **Quality Control**: Synthesized audio validated for clarity and naturalness

### Question-Passage Pairing
- Each passage audio is paired with multiple questions (avg ~4-5 questions per passage)
- Questions reference specific information within the spoken passage
- Answers are text spans that would appear in the original text passage

## Notes
- Audio files are sampled at **22.05 kHz** (train/dev) or **16 kHz** (test).
- Audio format varies by split: **MP3** for train/dev, **WAV** for test.
- This is a **textless spoken QA task** - models must process audio directly without transcription.
- The dataset is **based on SQuAD 1.1**, inheriting its question types and answer formats.
- **Multiple questions** can reference the same passage audio:
  - Train: Average of 4.61 questions per passage
  - Dev: Average of 5.08 questions per passage
  - Test: 1 question per passage
- Answer format is **extractive**: answers are text spans from the original passage.
- The `question` and `answer` fields contain text, but the model must answer based on the spoken `path` audio.
- Common question types include:
  - **Factual**: "What/Who/When/Where" questions
  - **Definitional**: "What is X?" questions
  - **Numerical**: Questions about quantities, dates, numbers
  - **Named Entity**: Questions about people, places, organizations
- The dataset is particularly valuable for:
  - **Textless speech processing**: Understanding speech without ASR
  - **Spoken language understanding**: Extracting information from audio
  - **Multi-modal reasoning**: Combining audio processing with QA reasoning
  - **End-to-end spoken QA**: Direct audio-to-text answer generation
- **Challenge aspects**:
  - Long-form audio context (30-60 seconds)
  - Multiple questions per passage requiring fine-grained understanding
  - No text transcriptions provided during inference
  - Diverse speaker characteristics from TTS synthesis
- The test set is notably smaller (171 questions) and uses 1 question per passage, likely for focused evaluation.
- IDs follow the format `{original_squad_id}_Q{question_number}` to maintain traceability to SQuAD source while ensuring uniqueness.

## Modeling Approaches

### 1. End-to-End Textless Models
```
Audio → Speech Encoder → Context Representation → Answer Generation
```
**Pros**: Truly textless, captures prosody
**Cons**: Requires large-scale pretraining

### 2. ASR-then-QA Pipeline
```
Audio → ASR → Text → Text QA Model → Answer
```
**Pros**: Leverage strong text QA models
**Cons**: Error propagation from ASR, loses acoustic information

### 3. Multi-modal Fusion
```
Audio → [Speech Features + ASR Text] → Fusion → Answer
```
**Pros**: Best of both approaches
**Cons**: More complex architecture

---

## Evaluation Metrics

Following SQuAD convention, models are typically evaluated using:
- **Exact Match (EM)**: Percentage of predictions that exactly match ground truth
- **F1 Score**: Token-level F1 between prediction and ground truth
- **Human Performance Comparison**: Compare against human answering accuracy
